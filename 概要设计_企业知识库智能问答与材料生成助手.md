# 企业知识库智能问答与材料生成助手概要设计

## 1. 文档说明

本文档根据《PRD_题目一_LangChain_企业知识库智能问答与材料生成助手.docx》整理，用于指导两天 AI 大赛 MVP 版本的设计、开发和演示。

本项目目标不是建设完整生产级企业知识库平台，而是实现一个可演示、可解释、可复用的 AI 应用 Demo，重点覆盖 LangChain、RAG、工具调用、结构化输出、来源引用、未知问题处理和材料生成能力。

## 2. 项目目标与范围

### 2.1 业务目标

- 降低员工查找制度、流程、FAQ、规范文档的时间成本。
- 支持基于企业知识库的自然语言问答，并展示可信来源。
- 基于检索到的知识内容生成可复用办公材料。
- 让参赛团队掌握 LLM 应用的基础工程链路：文档解析、切分、向量化、检索、提示词、工具调用、结构化输出、日志观测。

### 2.2 MVP 范围

MVP 必须完成：

- 文档上传或样例文档导入。
- 文本抽取、分块、向量化、索引构建。
- 基于知识库的问答。
- 回答来源引用，包括文件名、chunk_id、原文摘录。
- 知识库无依据时拒绝编造。
- 至少一种材料生成能力。
- 至少一个 LangChain 自定义工具。

MVP 建议完成：

- 多轮追问。
- 检索和模型调用日志展示。
- 知识库空间切换。
- 简单反馈能力，如点赞、点踩。

不纳入 MVP：

- 真实企业权限系统。
- 生产级知识库同步。
- 正式合同、法律意见、财务报告等高风险材料生成。
- 复杂审批流或真实 HR、OA、财务系统集成。

## 3. 待确认事项

以下事项会影响实现方案，建议开发前确认；如未确认，本文档给出默认选择。

| 事项 | 默认选择 | 需要确认的问题 |
| --- | --- | --- |
| Web 技术栈 | Streamlit 单体应用 | 是否必须使用 FastAPI + 前端框架？ |
| LLM 来源 | 管理员在系统配置中统一配置 OpenAI/Azure OpenAI/公司网关；环境变量可作为初始默认值 | 比赛现场可用的模型服务和 API Key 是什么？ |
| Embedding 模型 | 管理员统一配置，优先使用公司统一 embedding 或 OpenAI embedding | 是否允许调用外部 embedding 服务？ |
| API Key 配置方式 | 管理员一次配置，全局生效；普通用户不可查看或修改 | 是否需要支持多个模型配置方案并让用户选择？ |
| 文档格式 | Markdown、TXT、PDF、DOCX | 是否只要求支持其中两种格式即可？ |
| 向量库 | Chroma 本地持久化 | 是否有指定向量数据库？ |
| 材料类型 | 流程清单、FAQ | 演示时优先展示哪一种材料？ |
| 样例资料 | 使用模拟制度文档 | 是否提供统一样例资料包？ |
| 观测工具 | 本地日志表 | 是否要求接入 LangSmith？ |

## 4. 总体架构

### 4.1 架构视图

```text
用户浏览器
   |
   v
Streamlit Web UI
   |
   +-- 文档导入模块
   |      |
   |      v
   |   Loader -> Text Splitter -> Embedding -> Vector Store
   |
   +-- 模型配置模块
   |      |
   |      v
   |   Provider Config -> Chat Model Factory -> Embedding Factory
   |
   +-- 知识库问答模块
   |      |
   |      v
   |   Retriever -> Evidence Filter -> Prompt -> LLM -> Answer Formatter
   |
   +-- 材料生成模块
   |      |
   |      v
   |   Retriever -> Material Prompt -> LLM Structured Output -> Markdown Renderer
   |
   +-- 工具调用模块
   |      |
   |      v
   |   LangChain Tool: policy_search / calc_amount / mock_employee_lookup
   |
   +-- 日志与反馈模块
          |
          v
       SQLite / JSONL
```

### 4.2 推荐技术栈

| 分层 | 技术方案 | 说明 |
| --- | --- | --- |
| UI | Streamlit | 两天开发周期内效率最高，适合上传、问答、引用卡片、日志展示。 |
| 编排 | LangChain | 组合模型、Prompt、工具、检索器、结构化输出。 |
| 文档加载 | LangChain document loaders 或 Unstructured/PyMuPDF/python-docx | 支持 Markdown、TXT、PDF、DOCX。 |
| 文本切分 | RecursiveCharacterTextSplitter | 兼顾中文段落和固定长度切分。 |
| 向量库 | Chroma | 本地持久化，便于演示和重启恢复。 |
| 元数据存储 | SQLite 或 JSON | 记录文件、chunk、日志、反馈。 |
| LLM | 管理员页面配置 Chat Model | 支持配置供应商、模型名称、API 地址、温度、API Key；普通用户只使用已启用配置。 |
| Embedding | 管理员页面配置 Embedding Model | 支持配置供应商、模型名称、API 地址、API Key，优先选择中文效果较好的 embedding。 |

## 5. 核心流程设计

### 5.1 文档导入与索引流程

```text
上传/选择样例文档
   |
   v
校验文件格式与大小
   |
   v
按格式抽取文本
   |
   v
清洗文本并补充标题、页码、段落信息
   |
   v
切分 chunk
   |
   v
生成 embedding
   |
   v
写入 Chroma，并保存元数据
   |
   v
返回导入结果：成功文件数、失败文件、chunk 数、索引更新时间
```

分块策略：

- 默认 chunk_size：800 至 1200 中文字符。
- 默认 chunk_overlap：100 至 200 中文字符。
- 优先按标题、段落、列表切分；无法识别结构时使用固定长度切分。
- 每个 chunk 必须保留元数据：knowledge_base_id、source、file_name、chunk_id、title、page、created_at。

重复导入策略：

- 同一知识库中同名文件默认覆盖旧索引。
- UI 展示“覆盖导入”提示。
- 日志记录旧 chunk 删除数量和新 chunk 数量。

### 5.2 知识库问答流程

```text
用户问题 + 会话上下文
   |
   v
问题改写：将追问补全为独立问题
   |
   v
向量检索 top_k=5
   |
   v
证据过滤：相似度阈值 + 文本相关性判断
   |
   v
无有效证据？
   |-- 是 -> 返回未知问题模板
   |
   v
构造 RAG Prompt
   |
   v
LLM 生成结构化答案
   |
   v
格式化展示：直接回答、依据说明、来源、不确定性
   |
   v
记录日志
```

答案格式：

```text
直接回答：
...

依据说明：
...

来源：
[1] 文件名 - chunk_id - 原文摘录
[2] 文件名 - chunk_id - 原文摘录

不确定性：
...
```

未知问题处理：

- 当最高相似度低于阈值时，返回“知识库未找到依据”。
- 当检索片段与问题主题不一致时，返回“当前资料不足以回答”。
- Prompt 中明确禁止引入知识库之外的制度、金额、日期、审批人。
- 可建议用户补充需要的资料类型。

### 5.3 多轮追问流程

多轮追问使用 session_id 保存最近 N 轮对话。

处理方式：

- 将用户追问和最近 2 至 4 轮上下文交给 LLM 改写为独立问题。
- 使用改写后的问题检索知识库。
- 回答仍然必须基于新检索到的证据，不直接依赖模型记忆。

示例：

```text
上一轮：用户问“出差住宿费标准是多少？”
追问：那试用期员工也适用吗？
改写：差旅住宿费标准是否适用于试用期员工？
```

### 5.4 材料生成流程

```text
用户选择材料类型 + 输入主题
   |
   v
检索相关 chunk
   |
   v
证据不足判断
   |
   v
构造材料生成 Prompt
   |
   v
LLM 按结构化 schema 输出
   |
   v
渲染为 Markdown
   |
   v
展示来源并支持复制
```

MVP 优先支持两类材料：

| 类型 | 输入 | 输出 |
| --- | --- | --- |
| 流程清单 | 流程目标，如“申请差旅报销” | 步骤、负责人、所需材料、审批节点、风险提醒、来源 |
| FAQ | 主题，如“IT 账号申请” | 不少于 5 个问答，每条保留来源 |

结构化输出建议使用 Pydantic schema：

```python
class SourceRef(BaseModel):
    file_name: str
    chunk_id: str
    quote: str

class ProcessStep(BaseModel):
    step_no: int
    action: str
    owner: str | None
    required_materials: list[str]
    risk_tip: str | None
    sources: list[SourceRef]

class ProcessChecklist(BaseModel):
    title: str
    scope: str
    steps: list[ProcessStep]
    notes: list[str]
```

## 6. 功能模块设计

### 6.1 知识库管理模块

职责：

- 创建和切换知识库空间。
- 上传文档。
- 选择样例资料包。
- 触发索引构建或重建。
- 展示文件数、chunk 数、索引时间、失败原因。

核心状态：

- knowledge_base_id
- knowledge_base_name
- document_count
- chunk_count
- last_indexed_at
- status

### 6.2 文档处理模块

职责：

- 根据文件后缀选择 loader。
- 抽取正文、页码、标题等元数据。
- 统一清洗空行、页眉页脚、重复空白。
- 输出 LangChain Document 列表。

异常处理：

- 单个文件失败不影响其他文件导入。
- 失败文件在 UI 中展示文件名和错误类型。
- 解析为空文本时标记为失败。

### 6.3 检索模块

职责：

- 按 knowledge_base_id 做检索过滤。
- 支持 top_k 配置。
- 返回 chunk 文本、相似度、元数据。
- 可扩展 MMR、关键词检索或 rerank。

MVP 检索策略：

- 向量相似度检索 top_k=5。
- 相似度低于阈值时视为证据不足。
- 每次回答最多使用 3 至 5 条证据，避免 Prompt 过长。

### 6.4 问答生成模块

职责：

- 构造系统提示词和上下文。
- 调用 Chat Model。
- 生成固定格式答案。
- 输出引用和不确定性说明。

系统提示词核心约束：

```text
你是企业知识库助手。你只能基于给定资料回答。
如果资料中没有明确依据，必须说明“知识库未找到依据”，不能编造。
回答中的制度、金额、日期、流程、审批人必须能在来源中找到。
每个关键结论都要关联至少一个来源编号。
```

### 6.5 材料生成模块

职责：

- 根据材料类型选择模板。
- 基于检索证据生成材料。
- 使用结构化输出保证字段完整。
- 渲染为 Markdown 并展示来源。

材料生成安全规则：

- 不生成知识库之外的承诺。
- 不自动补充未知金额、日期、审批角色。
- 对资料不足的字段标注“资料未说明”。

### 6.6 工具调用模块

MVP 至少实现一个自定义工具，建议实现 `policy_search`。

工具定义：

| 工具 | 入参 | 出参 | 用途 |
| --- | --- | --- | --- |
| policy_search | query, knowledge_base_id, top_k | source_chunks | 让 agent 按需检索制度片段。 |

可选工具：

| 工具 | 用途 |
| --- | --- |
| calc_amount | 计算差旅报销金额、超标金额等简单数值。 |
| classify_policy_type | 判断问题属于考勤、差旅、IT、研发流程等类型。 |
| mock_employee_lookup | 使用模拟员工信息判断员工类型，如正式、试用期、外包。 |

工具调用过程需在页面或日志中展示：

- 工具名称。
- 输入参数。
- 返回片段数量或计算结果。
- 调用耗时。

### 6.7 日志与反馈模块

记录内容：

- session_id
- question
- rewritten_question
- knowledge_base_id
- retrieved_chunks
- tool_calls
- answer
- material_type
- latency_ms
- feedback
- created_at

展示方式：

- 页面展示最近 20 条问答日志。
- 每条日志可展开查看检索片段、相似度、工具调用。
- 点赞/点踩记录到本地存储。

### 6.8 模型配置模块

职责：

- 管理员在系统配置页面中统一配置 Chat Model 和 Embedding Model。
- 支持选择模型供应商，如 OpenAI、Azure OpenAI、公司统一网关、自定义 OpenAI-compatible 接口。
- 支持配置模型名称、Base URL、API Key、温度、超时时间、最大输出长度。
- 支持配置 embedding 模型名称、Base URL、API Key、向量维度说明。
- 支持“测试连接”按钮，验证模型和 embedding 是否可用。
- 支持“启用配置”按钮，使所有普通用户后续文档索引、问答和材料生成使用当前全局配置。
- 支持“恢复环境变量默认配置”。
- 普通用户不需要配置 API Key，也不能查看完整 API Key。

配置生效规则：

- 管理员配置优先级高于环境变量；环境变量用于首次启动或兜底。
- API Key 由管理员一次配置，全局生效。
- API Key 持久化时必须加密或使用部署环境的密钥管理能力；MVP 可使用本地 `.env` 或仅管理员可访问的本地配置文件。
- 普通用户页面只展示模型供应商、模型名称、当前配置状态，不展示 API Key。
- 非敏感配置可保存到本地 SQLite 或 JSON，如 provider、model_name、base_url、temperature、top_k。

管理员访问控制：

- MVP 可使用 `ADMIN_PASSWORD` 或 `ADMIN_TOKEN` 环境变量保护管理员系统配置页。
- 管理员 API 必须校验管理员身份，普通用户请求应返回无权限。
- 生产化版本建议接入公司 SSO/RBAC，将模型配置权限限定给平台管理员或系统管理员。

配置变更影响：

- Chat Model 配置变更后，新问答和新材料生成立即生效。
- Embedding 配置变更后，已有向量索引不自动兼容，页面必须提示用户重新构建索引。
- top_k、similarity_threshold、temperature 等参数变更后立即生效。
- 管理员变更模型配置需记录操作日志，包括操作人、变更字段、变更时间和连接测试结果。

## 8. API 设计

如果采用 Streamlit 单体应用，以下 API 可作为内部函数边界；如果升级为 FastAPI，可直接暴露为 HTTP 接口。

| 接口 | 方法 | 说明 | 关键字段 |
| --- | --- | --- | --- |
| /api/knowledge-bases | GET | 获取知识库列表 | - |
| /api/model-config | GET | 获取当前启用模型配置，普通用户不返回 API Key | - |
| /api/admin/model-config | PUT | 管理员更新全局模型配置 | chat_provider, chat_model_name, chat_base_url, chat_api_key, embedding_model_name, embedding_api_key |
| /api/admin/model-config/test | POST | 管理员测试模型和 embedding 连接 | model_type, provider, model_name, base_url, api_key |
| /api/admin/model-config/enable | POST | 管理员启用指定模型配置 | config_id |
| /api/documents | POST | 上传并索引文档 | files, knowledge_base_id |
| /api/chat | POST | 知识库问答 | session_id, question, knowledge_base_id, top_k |
| /api/generate | POST | 生成材料 | material_type, topic, knowledge_base_id, source_ids |
| /api/sources/{id} | GET | 获取引用原文 | source_id |
| /api/logs | GET | 查看日志 | session_id, limit |
| /api/feedback | POST | 提交反馈 | log_id, feedback |

### 8.1 问答请求示例

```json
{
  "session_id": "s_001",
  "knowledge_base_id": "kb_employee",
  "question": "出差住宿费标准是多少？如果超标怎么审批？",
  "top_k": 5
}
```

### 8.2 问答响应示例

```json
{
  "answer": "根据差旅报销制度，住宿费标准按城市级别执行；如超标，需要提前提交说明并经过部门负责人审批。",
  "sources": [
    {
      "source_id": "chunk_001",
```

## 9. 页面设计

### 9.1 页面布局

采用单页多 Tab：

- 管理员系统配置：配置 Chat Model、Embedding Model、API Key、检索参数和测试连接。
- 知识库：选择知识库、上传文档、导入样例、查看索引状态。
- 智能问答：输入问题、展示答案、展示来源引用、展示工具调用。
- 材料生成：选择材料类型、输入主题、展示生成结果、复制 Markdown。
- 日志评估：查看最近问题、检索结果、反馈、响应耗时。

### 9.2 关键交互

管理员系统配置：

- 管理员输入管理员密码或通过管理员入口进入系统配置页。
- 管理员选择模型供应商，填写模型名称、Base URL、API Key。
- 管理员选择或填写 embedding 模型配置。
- 管理员调整全局默认 temperature、top_k、similarity_threshold。
- 管理员点击“测试连接”，页面分别验证 Chat Model 和 Embedding Model。
- 管理员点击“启用配置”，后续所有用户的索引、问答、材料生成使用新配置。
- 如果 embedding 配置发生变化，页面提示“需要重新构建索引”。
- 普通用户不进入管理员配置流程，只能在页面看到当前模型名称、知识库状态和可用功能。

知识库导入：

- 用户上传文件后点击“构建索引”。
- 页面展示导入进度和结果摘要。
- 失败文件可展开查看原因。

问答：

- 用户输入问题并提交。
- 页面显示“正在检索资料”和“正在生成回答”。
- 答案下方展示引用卡片。
- 引用卡片可展开查看原文摘录、文件名、chunk_id、页码。

材料生成：

- 用户选择“流程清单”或“FAQ”。
- 输入主题。
- 系统先检索来源，再生成材料。
- 结果以 Markdown 展示，可复制。

日志：

- 展示最近问答列表。
- 支持展开查看 retrieved_chunks、similarity、tool_calls。
- 支持点赞/点踩。

## 10. 关键提示词设计

### 10.1 问答 Prompt

```text
你是企业知识库智能问答助手。

要求：
1. 只能根据“知识库资料”回答。
2. 如果资料不足，必须回答“知识库未找到依据”或“当前资料不足以回答”。
3. 不得编造政策、金额、日期、审批人、流程节点。
4. 每个关键结论必须引用来源编号。
5. 回答使用以下结构：
   - 直接回答
   - 依据说明
   - 来源
   - 不确定性

用户问题：
{question}

知识库资料：
{context}
```

### 10.2 材料生成 Prompt

```text
你是企业办公材料生成助手。

你需要基于给定知识库资料生成 {material_type}。
不得使用资料外的信息补全政策内容。
如果某个字段资料没有说明，请写“资料未说明”。
所有步骤、规则、注意事项都要保留来源。

主题：
{topic}

知识库资料：
{context}
```

### 10.3 问题改写 Prompt

```text
请根据最近对话，将用户追问改写成一个可以独立检索知识库的问题。
只输出改写后的问题，不要回答。

最近对话：
{chat_history}

用户追问：
{question}
```

## 11. 非功能设计

### 11.1 性能

- 样例知识库不超过 30 份文档。
- 普通问答目标响应时间：15 秒内。
- 材料生成目标响应时间：30 秒内。
- 向量检索应在 1 秒内完成。

### 11.2 稳定性

- 文档导入失败不影响应用启动。
- 单文件解析失败不影响其他文件。
- 模型调用失败时展示可理解错误信息。
- 向量库为空时提示用户先导入文档。

### 11.3 安全

- API Key 由管理员统一配置，普通用户不可查看、不可修改、不可导出。
- API Key 不写入代码，不以明文提交到代码仓库。
- MVP 可通过 `.env` 或管理员配置文件加载 API Key；如落库必须加密保存或保存密钥引用。
- 普通用户接口和页面只返回模型供应商、模型名称、配置状态，不返回 API Key。
- 管理员系统配置页和 `/api/admin/*` 接口必须做管理员鉴权。
- 管理员模型配置变更需要记录操作日志。
- `.env`、上传文件、向量库目录加入 `.gitignore`。
- 演示资料必须使用模拟数据或脱敏数据。
- Prompt 明确禁止输出资料外的敏感承诺。

### 11.4 可解释性

- 每个回答必须展示来源。
- 日志中记录检索片段和工具调用。
- 未知问题明确说明没有依据。

### 11.5 可配置性

建议配置项：

```text
CHAT_PROVIDER
MODEL_NAME
MODEL_BASE_URL
MODEL_API_KEY
EMBEDDING_PROVIDER
EMBEDDING_MODEL_NAME
EMBEDDING_BASE_URL
EMBEDDING_API_KEY
CHUNK_SIZE
CHUNK_OVERLAP
TOP_K
SIMILARITY_THRESHOLD
TEMPERATURE
TIMEOUT_SECONDS
MAX_TOKENS
ADMIN_PASSWORD
ADMIN_TOKEN
```

页面配置范围：

- 管理员可页面配置：provider、model_name、base_url、api_key、temperature、top_k、similarity_threshold、timeout_seconds、max_tokens。
- 普通用户可页面调整：本次问答的 top_k、材料类型、知识库选择等业务参数；不可配置模型服务和 API Key。
- 可持久化保存：provider、model_name、base_url、temperature、top_k、similarity_threshold、timeout_seconds、max_tokens。
- API Key 持久化方式：不保存明文；优先使用 `.env`、密钥管理服务或加密后的本地配置。
- embedding 配置变化后必须提示重新索引。

## 13. 两天开发计划

### 第 1 天上午：基础工程与索引链路

- 初始化项目结构。
- 实现管理员模型配置表单，支持 Chat Model、Embedding Model、Base URL、API Key、temperature、top_k。
- 实现模型和 embedding 连接测试及全局启用配置。
- 准备样例数据。
- 实现 Markdown/TXT/PDF/DOCX 中至少两种 loader。
- 实现文本切分、向量化、Chroma 入库。

产出：可构建知识库索引。

### 第 1 天下午：RAG 问答与引用展示

- 实现检索器。
- 实现问答 Prompt。
- 实现答案格式化。
- 实现来源引用卡片。
- 准备 10 条测试问题。

产出：可完成带来源的知识库问答。

### 第 2 天上午：材料生成、工具调用、多轮追问

- 实现流程清单或 FAQ 生成。
- 实现 `policy_search` 自定义工具。
- 实现会话上下文和追问改写。
- 实现未知问题模板。

产出：完整演示闭环。

### 第 2 天下半天：打磨、日志与演示

- 增加日志和反馈。
- 完善错误提示。
- 补充 README、启动方式、样例问题。
- 准备 5 分钟演示脚本。

产出：最终提交物。

## 14. 演示脚本

1. 展示知识库列表和样例文档，说明已导入员工手册、差旅制度、研发流程、IT FAQ、AI 使用规范。
2. 展示管理员系统配置页，说明管理员已统一配置模型、Base URL、API Key、temperature、top_k，并执行一次连接测试和启用操作。
3. 提问“出差住宿费标准是多少？如果超标怎么审批？”，展示答案和引用。
4. 追问“试用期员工也适用吗？”，展示多轮追问和来源。
5. 提问“明年公司是否增加年假？”，展示拒绝编造。
6. 输入“申请差旅报销”，生成流程清单。
7. 展示日志页面，说明检索片段、工具调用、最终输出。
8. 简述技术链路：管理员模型配置 -> loader -> splitter -> embedding -> vector store -> retriever -> prompt/model -> tools -> structured output。

## 15. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 模型服务不稳定 | 演示失败 | 支持切换模型配置，准备缓存样例结果。 |
| 中文检索效果不佳 | 回答不准 | 调整 chunk_size，增加关键词检索或 MMR。 |
| PDF/DOCX 解析质量差 | 来源不完整 | 优先使用 Markdown/TXT 样例资料，PDF/DOCX 作为增强。 |
| 未知问题仍被模型发挥 | 可信度下降 | 加强阈值判断和 Prompt 约束，低置信度直接走模板。 |
| 两天时间不足 | 功能不闭环 | 优先保证文档导入、问答、引用、未知问题、材料生成。 |
